# OmniAgent v3 Improvement Plan - UPDATED

## Executive Summary

**Critical Finding**: MLX (Apple's Metal framework) is NOT functional on Asahi Linux despite successful pip installation. The `libmlx.so` shared library cannot be loaded.

**Root Cause**: MLX is designed for macOS Metal API, not Linux Vulkan/Mesa. While the Python package installs on aarch64 Linux, the underlying C++ library requires macOS-specific Metal frameworks.

## Revised Strategy

### Option 1: Continue Vulkan Optimization (Current Path)
**Status**: ✅ WORKING - 22 TPS on Llama-8B Q4_K_M
**Gap**: 3-10x slower than MLX on macOS
**Action**: Optimize existing Vulkan backend

**Priority Tasks**:
1. **Graph Caching** (HIGH) - Eliminate 3ms/token rebuild overhead
   - Pre-build ggml graph once at model load
   - Use `ggml_gallocr` for static allocation
   - Target: <0.5ms graph build time
   - Expected gain: 22 TPS → 30-33 TPS

2. **Memory Residency** (HIGH) - Reduce CPU↔GPU copies
   - Pre-allocate KV cache on GPU
   - Keep weights in Vulkan device memory
   - Use pinned memory mappings
   - Expected gain: 20-40% latency reduction

3. **Custom Compute Shaders** (MEDIUM) - Replace generic `torch.mm`
   - Study llama.cpp Vulkan shaders
   - Implement tiled GEMM with shared memory
   - Fuse gate_up + SiLU + down into single dispatch
   - Expected gain: 30-60% GPU utilization (vs current 11%)

### Option 2: Hybrid CPU+Vulkan (Fallback)
**Status**: ⚠️ RESEARCH NEEDED
**Concept**: Use Vulkan for memory, CPU for compute
- Store weights in Vulkan buffers (unified memory)
- Map to CPU address space
- Use Apple AMX/NEON BLAS (OpenBLAS/Accelerate)
- Zero-copy from GPU-allocated memory

**Expected**: CPU BLAS speed (~50-70 TPS on 0.5B)
**Effort**: 30-50 hours

### Option 3: Boot macOS for MLX (Practical)
**Status**: ✅ WORKS - 80-120 TPS on 0.5B
**Trade-off**: Defeats "Linux on Apple Silicon" research goal
**Use Case**: Production deployments where performance > research
